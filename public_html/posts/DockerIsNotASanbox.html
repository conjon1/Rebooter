Docker is not a Sandbox

One of the biggest misconceptions I see about Docker is that it’s somehow sandboxed—a place where you can safely test programs and services in isolation.

This is simply not true.

In reality, Docker is just sophisticated plumbing on top of your system. It uses root privileges, a background daemon, and a set of Linux kernel features like namespaces and cgroups to keep the container contained—but only by convention, not by isolation guarantees.

Docker makes development frictionless, but it was never designed to be a fortress. Treating it like one is a common and dangerous mistake.

In this post, we’ll explore how Docker actually works from an architectural standpoint, expose the limits of its “sandbox,” and show a few examples of how containers can break free—plus best practices to mitigate those threats.

How Docker Works

“Docker is a set of platform as a service (PaaS) products that uses operating-system-level virtualization to deliver software in packages called containers.” — Wikipedia

Docker isn’t magic—it’s a system built on top of Linux primitives. Understanding those primitives is key to knowing why it isn’t a sandbox.

The purpose of Docker

Docker is operating-system-level virtualization, not hardware-level virtualization.

That distinction matters. It means that all containers on a Linux host share the same kernel. They aren’t virtual machines with their own hardware abstraction—they’re processes running on the same kernel, isolated by rules , not by hardware.

That’s what makes Docker so lightweight and fast: starting a container is like starting a process. It’s perfect for packaging and shipping software along with its dependencies.

But that shared kernel also means containers are only as secure as the kernel itself. A vulnerability in the kernel, or in Docker’s configuration, can give a container direct access to the host.

Overview of Docker’s Architecture

Docker’s architecture is built around three core components that work together to make containers feel isolated:

  1. Docker Daemon (dockerd) – The engine running on your host. It manages images, containers, networks, and volumes. It runs as root by default, which means every container operation ultimately touches privileged parts of your system.
  2. Container Runtime (containerd and runc) – The tools that actually create containers. They talk directly to the Linux kernel, using system calls to spawn processes inside isolated namespaces and apply cgroups for resource control.
  3. Images and Filesystems – Containers are built from layered filesystems (like OverlayFS). Each layer is read-only, and a writable layer is added on top when the container runs. It feels like a clean filesystem—but under the hood, it’s a collection of diffs and mount points.

At a high level, Docker provides a user-friendly interface to these low-level kernel features. You type docker run, and Docker orchestrates namespaces, control groups, mounts, and network bridges for you.

It’s elegant. It’s efficient. But it’s not isolated.

How the Illusion Works

Docker’s magic trick is the illusion of separation. Here’s how the main components play their part:

  • Namespaces – Create isolated views of the system: process IDs, filesystems, users, networks. Each container thinks it’s alone, but all share the same kernel.
  • Cgroups – Limit CPU, memory, and I/O usage. They prevent one container from hogging resources, but not from exploiting the kernel.
  • OverlayFS – Creates the illusion of a separate root filesystem. Changes made in a container don’t affect the base image—but they still live on the same host storage.
  • Bridged Networking – Docker’s network stack uses virtual Ethernet bridges (like docker0) to connect containers. It’s convenient, but not inherently secure.

Docker’s isolation stops at the kernel boundary. It doesn’t emulate hardware. It doesn’t create true virtual machines.

It’s a shared playground, not a sealed vault.

Threat Modeling Docker’s Architecture — how it works, how it breaks, how to stop it

We’ll walk component by component. For each: a short explainer of how it works, the concrete ways it is abused in the wild, and clear, actionable mitigations you can use today.

Docker daemon (dockerd) — the one process that changes everything

How it works

  • dockerd is the service that manages images, containers, networks and volumes.
  • It exposes a UNIX socket (/var/run/docker.sock) and an HTTP API. The daemon usually runs as root and performs privileged operations on behalf of users or services.
How it’s abused
  • Socket abuse: If a container or user can write to /var/run/docker.sock they can talk to the daemon as if they were admin. The Docker API can create new containers, mount host paths, set --privileged, and run arbitrary commands — i.e., full host control.
  • Misconfigured access: Adding untrusted users to the docker group effectively gives them root-equivalent powers because they can control the daemon.
  • Remote exposure: Exposing the API over TCP (even with TLS misconfigured) opens the same attack surface to the network.

Why it’s powerful as an attack vector

  • The daemon is a privileged choke point. Everything that runs through it gets installed/started with host-level hooks (mounts, network setup, capabilities).
Safe mitigations
  • Never mount the Docker socket into untrusted containers.
  • Don’t add untrusted users to the docker group. Treat membership like sudo.
  • Prefer rootless Docker for developer machines and CI runners where full root access is unnecessary.
  • Avoid exposing the daemon over TCP. If you must, require mTLS and strict ACLs.
  • Audit who can call Docker APIs (CI agents, runners, orchestration services).

Container runtime (containerd, runc) — where the kernel gets invoked

How it works

  • containerd is the longer-lived runtime manager; runc is a minimal runtime that performs the syscalls to create namespaces, cgroups, mounts and eventually execute your process.
  • runc essentially does the clone(2)/unshare(2)/setns(2) plumbing and then execs the process in the new namespaces.

How it’s abused

  • Runtime vulnerabilities: Bugs in runc or the runtime glue (race conditions, file-descriptor leaks, incorrect path handling) can be used to trick the runtime into mounting or writing host paths, overwriting runtime binaries, or gaining elevated privileges.
  • Supply-chain/CI build attacks: If an attacker can control image builds or builder environments, they can craft build-time operations that abuse the runtime (e.g., manipulating working directories or file descriptors) to influence host state.

Concrete mechanism (conceptual)

  • Attackers do not need to "hack the kernel" necessarily — they can exploit how the runtime calls the kernel. For example, a crafted sequence that abuses a runc race or a file descriptor handed to runc can cause it to operate on a host path instead of the intended container path.

Safe mitigations

  • Keep runtimes patched (containerd, runc) — these get CVEs and fixes frequently.
  • Reduce attack surface with seccomp profiles and LSMs (AppArmor/SELinux).
  • Use intermediate sandboxes like gVisor or Kata for untrusted code — they insert an extra boundary between user code and the kernel.
  • Harden build pipelines: run builds in isolated VMs, restrict build-time mounts, and avoid running privileged build containers.

Images & layered filesystems (OverlayFS) — what you pull is what you run

How it works

  • Images are stacks of immutable layers; when you run a container a writable top layer is added and mounts are configured to assemble the final root filesystem. OverlayFS merges layers efficiently.
  • Registries (Docker Hub, private registries) store images and metadata.

How it’s abused

  • Malicious images / supply chain attacks: An attacker publishes an image containing malware, trojans, or outdated, vulnerable libraries.
  • Typosquatting or latest-blind pulls: Relying on :latest or unverified public images lets you pull compromised content accidentally.
  • Image metadata tricks: Attackers can hide malicious actions in entrypoints or build steps that run at startup.

Why it’s dangerous

  • Once you docker run an image you trust the image’s contents and startup behavior. The image can create privileged processes, write to mounted volumes, or open network callbacks.

Safe mitigations

  • Pin image digest/versions rather than using :latest.
  • Sign images and use Docker Content Trust or a private signed registry.
  • Scan images in CI with Trivy/Clair/Snyk to detect known vulnerabilities.
  • Use minimal base images and rebuild images from minimal sources you control.

Namespaces & Cgroups — the illusion of separation

How it works

  • Namespaces create per-container views (PID, mount, network, user, UTS, IPC). A process in a PID namespace sees its own PID tree.
  • Cgroups cap and account for resource usage (CPU, memory, blkio, etc.).

How it’s abused

  • Kernel exploits: namespaces are a visibility boundary; they are not a security boundary against kernel bugs. Privilege escalation bugs in the kernel (or use-after-free, pipe/race issues) can let a process in a namespace affect the global kernel state.
  • Misuse of user namespaces: when user namespaces are not used, processes inside often map to root on the host (UID 0 inside = UID 0 on host in many setups). That makes container root extremely powerful.

Concrete mechanism (conceptual)

  • A kernel vulnerability that allows arbitrary memory write or improper handling of file descriptors will not respect “namespace separation” — the attacker is running on the same kernel and can manipulate global kernel structures or kernel-managed files (like /proc), leading to escapes.

Safe mitigations

  • Use user namespaces where feasible so container root is not host root.
  • Avoid --privileged which removes many namespace protections and adds all capabilities.
  • Limit capabilities with --cap-drop ALL and only add what is needed with --cap-add.
  • Keep kernels patched; adopt LTS kernels with quick patching workflows.

Mounts and bind mounts — the convenience that bites

How it works

  • Docker allows you to mount host paths into containers (-v /host/path:/container/path) and mount devices (--device). This is essential for persistence and access to hardware.

How it’s abused

  • Host filesystem access: Mounting / or /var into a container gives the container direct read/write on host data.
  • Device mounts: Mapping in devices (e.g., /dev/kvm) can allow code to interact with hardware in ways that allow privilege escalation or kernel attack surfaces.

Why it’s dangerous

  • Bind mounts blur the line between the host and the container; they let the container bypass the "illusion" of a separate filesystem.

Safe mitigations

  • Avoid host-root mounts unless absolutely necessary. Use managed volumes instead.
  • Mount read-only where possible (:ro) and restrict which host paths are exposed.
  • Audit CI scripts and admin helpers that mount the host for debugging — these are common accidental backdoors.

Networking — the shared neighborhood

How it works

  • Docker by default creates a bridge network (docker0) and attaches containers to it so they can talk to one another and to the outside world.
  • You can create user-defined networks, overlay networks (in swarm/k8s), host networking, or macvlan.

How it’s abused

  • Lateral movement: If an attacker compromises one container they can scan and reach other containers on the same network.
  • ARP/packet spoofing on a flat bridge if networks are not isolated.
  • Exposed services: Containers often expose ports; forgotten open services become attack vectors.

Safe mitigations

  • Use user-defined networks and segment workloads by security group / network policy.
  • Use network policies (in Kubernetes) or firewall rules on the host to limit cross-container traffic.
  • Don’t run sensitive services on default bridge; use host-networking only when explicitly needed and understood.

Practical hardening checklist (commands & patterns)

  • Run containers with least privilege:
docker run --rm
--cap-drop ALL
--security-opt no-new-privileges:true
--security-opt seccomp=/path/to/seccomp.json
--read-only
-v /data/app: /app:ro
your/image:tag
  • Use seccomp and LSMs (AppArmor/SELinux). Start with Docker’s default seccomp and lock it down if necessary.
  • Scan images in CI:
  • Add Trivy or equivalent to your pipeline and fail builds on critical findings.
  • Rootless Docker for dev and CI where possible.
  • Restrict dockerd access — don’t expose the socket, don’t add non-admins to docker group.
  • Use immutable image digests: your/image@sha256:<digest>.

Final, practical thoughts

If you’re trying to protect a host from accidental or malicious containers: treat Docker as a tool that can run untrusted code but must not be trusted by default. Build layers of protection:

  • secure the daemon and access to it,
  • harden the runtime (patch, LSM, seccomp),
  • secure the supply chain (sign, scan, pin),
  • minimize host exposure (no host mounts, avoid privileged),
  • isolate networking.

If you’re running code you don’t fully control, consider the extra cost of running it inside a VM or a secondary isolation technology (gVisor/Kata) instead of bare Docker.
