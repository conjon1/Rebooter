My Homelab Journey: From Mini PCs to Kubernetes with K3s

From a few mini PCs to a fully functional Kubernetes cluster – my homelab journey using Ubuntu and K3s has been full of learning experiences (and a few headaches!).

Motivation: The Pull of Automation

I've always been drawn to automation and building effective, interconnected systems. This interest was likely first sparked by games like Minecraft with its automatic farms, and later solidified by titles like Factorio and Mindustry. In Factorio, in particular, you design intricate, interconnected factories to automate production towards launching a rocket – it’s incredible dopamine, like candy for my brain. This love for building complex systems naturally led me to explore the world of container orchestration with Kubernetes.

Why K3s?

When I discovered K3s, I was immediately interested. It seemed like a great starting point: providing a solid Kubernetes foundation that was relatively easy to set up but didn't abstract away too much, unlike options like MicroK8s from Canonical. I didn't want it to be overly simplified; I wanted the challenge of navigating some complexities myself to gain a deeper understanding of the technology. In the future, I might consider migrating the cluster to Talos Linux for its immutable, API-driven approach, likely paired with standard K8s (Kubernetes).

The Hardware

For my Kubernetes homelab, I chose 3x Beelink EQ12 mini PCs. These compact machines offered the perfect blend of low power consumption and sufficient processing power for my current and future experiments.

Specifications

  • Processor: Each unit features an efficient Intel processor.
  • RAM: 24GB DDR5 RAM per unit.
  • Storage: 500GB NVMe SSD per unit.

Low Power Consumption

Since the cluster runs 24/7, minimizing energy usage was crucial for keeping operating costs manageable.

Future-Proofing (to a degree)

While not high-end servers, these mini PCs provide enough resources to run various Kubernetes workloads and explore different configurations.

Processing Power

Each machine provides 10 Cores / 12 Threads. Across the three nodes, this gives a total of 30 cores and 36 threads, which is a decent amount of processing power for this learning project.

Node Count

I opted for three nodes to build a more resilient setup. This allows me to experiment with concepts like high availability, although with K3s's embedded database, it's often more about having dedicated worker nodes alongside the control plane.

Networking

The nodes are connected using a basic Gigabit Ethernet switch.

The Operating System: Ubuntu with a Security Twist

Initially, I tried running Talos Linux, but I quickly realized I was a bit out of my depth. Getting the system configured exactly as I wanted proved challenging (mostly user error, honestly!). I also considered Arch Linux, but having used it as my primary desktop OS for years, I anticipated potential maintenance headaches down the line when obscure problems inevitably arise.

So, I turned my attention to Debian-based systems, and Ubuntu immediately came to mind. I have experience with the apt package manager, and Debian/Ubuntu are known for being significantly more stable than Arch for server environments.

For added security, I opted for LUKS full-disk encryption on each node. The initial drawback is needing to enter the decryption passphrase manually on each machine during boot, which is cumbersome. As a temporary workaround, before I potentially set up automated network-bound unlocking (perhaps using a Tang server), I'm using a Bluetooth keyboard capable of switching between the three machines. It's a simple but effective fix for now!

Getting Hands-On: Installing K3s on Ubuntu

With the hardware chosen and Ubuntu Server 24.04.2 LTS installed on my three "honey" nodes (yes, sticking with the bee theme thanks to the Beelink hardware), it was time to dive into the K3s installation.

Disabling the Firewall

Ubuntu Server comes with ufw (Uncomplicated Firewall) enabled by default. I SSH'd into each node and disabled it with:

sudo ufw disable
sudo systemctl disable ufw

Installing the K3s Control Plane (on honey1)

I started by installing the K3s server component on honey1:

curl -sfL https://get.k3s.io | sh -

After installation, I retrieved the node token from:

sudo cat /etc/rancher/k3s/k3s.yaml

Joining the Worker Nodes (honey2 & honey3)

Next, I SSH'd into honey2 and honey3 and used the installation script again, providing the control plane's URL and token:

curl -sfL https://get.k3s.io | K3S_URL=https://<honey1-ip>:6443 K3S_TOKEN=<your-token> sh -

Setting up kubectl on Your Host Machine

  1. Copy K3s Config: Copy the entire contents of the /etc/rancher/k3s/k3s.yaml file from honey1.
  2. Create Local Config: On my Arch Linux workstation, I created the config file: 
    mkdir -p ~/.kube
nano ~/.kube/config
  3. Paste and Modify: I pasted the copied content and changed the server address from https://127.0.0.1:6443 to https://<honey1-ip>:6443.

After saving, I could run kubectl get nodes and see all three nodes ready!

Monitoring with Prometheus & Grafana

With the cluster running, the next step was setting up a monitoring stack using Prometheus and Grafana via a Helm chart.

Installation via Helm

First, I added the Prometheus community Helm repository:

helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update

Then, I installed the stack into its own namespace:

helm install prometheus-stack prometheus-community/kube-prometheus-stack \
  --namespace=prometheus-stack \
  --create-namespace

Making Access Easier: NodePort and Ingress

To access Grafana from my local network, I created two Kubernetes objects:

  1. A Service of type NodePort to expose the Grafana deployment.
  2. An IngressRoute for Traefik to route a hostname like grafana.local to the service.

Here's the YAML I used:

# grafana-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: grafana-service
  namespace: prometheus-stack
spec:
  type: NodePort
  selector:
    app.kubernetes.io/name: grafana
  ports:
    - protocol: TCP
      port: 3000
      targetPort: 3000
      nodePort: 30001
# grafana-ingressroute.yaml
apiVersion: traefik.containo.us/v1alpha1
kind: IngressRoute
metadata:
  name: grafana-ingressroute
  namespace: prometheus-stack
spec:
  entryPoints:
    - web
  routes:
    - match: Host(`grafana.local`)
      kind: Rule
      services:
        - name: grafana-service
          port: 3000

After applying these files with kubectl apply -f ., I could access the Grafana dashboard at http://grafana.local in my browser. Much easier!
